## v5.1 Phase C — Scenario historical mode + setup notes

**Goal**: a freshly-launched scenario should be useful for a demo or
investigation **immediately**, with the whole attack story already in
the past and aged into realistic timestamps. Also: a scenario should
tell the operator exactly which collectors to configure for it to play
out fully.

Two coordinated changes, shipped together:

### C.1 — Historical mode (default stays Realtime)

New scenario field `mode: "realtime" | "historical"` (default
`realtime` — preserves today's UX bit-for-bit).

**Realtime** (current): scheduler clock starts at launch, phases
activate over wall-clock, events fire as collectors pull, timestamps
inherit from the live mix.

**Historical** (new): at launch, **all** events for the entire
duration are pre-computed with timestamps spanning
`[now - duration, now]` and queued in a per-source, per-scenario
backlog. The scenario is marked `completed` immediately; no scheduler
thread runs. The next collector pull on each source drains its slice
of the backlog ahead of the live event mix.

**Data model**

- Scenario JSON gains:
  - `mode: "realtime" | "historical"` (default `"realtime"`).
  - `events_per_phase: int | null` (optional override; default
    computed per-phase as `max(1, phase_duration_s // periodicity)`).
  - `owner_id: str | null` and `visibility: "private" | "public"`
    (default `"private"`). Today scenarios are effectively global; we
    add ownership now so historical drain can honor per-caller
    visibility the same way live detection rules do.
- New on-disk artifact `data/attack_scenarios/<sid>_backlog.jsonl`,
  one line per pre-staged event:
  `{ "_source": "okta", "_owner_id": "...", "_visibility": "private",
     "_ts": 1729..., "_scenario_id": "...", "_phase_id": "...",
     ...event payload with timestamp field forced to _ts... }`.
- Sidecar `data/attack_scenarios/<sid>_backlog.idx.json`:
  `{ "okta": next_byte_offset, "aws_cloudtrail": ..., ... }`. Drain is
  O(events drained) — no full-file scan per pull.

**Pre-staging algorithm (`pre_stage_historical_events(scenario_id)`)**

For each phase:

1. Compute the absolute window
   `[t0, t1] = [now - duration + offset, now - duration + offset + phase_duration]`.
2. Compute `event_count`:
   - If `phase.events_per_phase` is set → use it verbatim.
   - Elif scenario-level `events_per_phase` is set → use it for every phase.
   - Else for time-based rules (`periodicity > 100`):
     `event_count = max(1, phase_duration_s // periodicity)`.
   - Else (count-based): `event_count = max(1, phase_duration_s // 60 * APIGENIE_SCN_ASSUMED_RATE_PER_MIN)`
     (env-var default `2`).
3. Pick `event_count` timestamps uniformly inside `[t0, t1]`. For each:
   - Generate a base event for that source via the existing source
     module's `_generate_*` helper (we'll add a thin
     `sources.<src>.generate_one(ctx)` helper or fall back to the
     module-level template path used by `inject_detection_events`).
   - Apply the phase's `field_overrides` plus `attack.id`, `phase.id`.
   - Force the source-specific timestamp field to the chosen `_ts`
     (per-source map: `okta → published`, `aws_cloudtrail →
     eventTime`, `cisco_duo → timestamp`, etc.).
   - Append a backlog line.
4. Update `attack_scenarios._scenario_event_logs` ring buffer
   immediately so the scenario card shows the full event count + the
   "preview" rows at launch.

**Drain (`drain_historical_backlog(source, max_items=None)`)**

- Resolves caller via `profiles.get_current_user()` (same ContextVar
  `auth.py` sets when an inbound API token matches a registered
  identifier).
- Reads the sidecar offset for *source*, scans forward in
  `<sid>_backlog.jsonl` only for lines tagged with that source,
  filters by visibility (public OR owner == caller_id OR caller is
  admin), advances the offset atomically, returns the events.
- All-at-once by default per (source, caller) on the first eligible
  pull — that's the demo-immediacy use case. `max_items` reserved for
  future capping.
- Hooked at the top of `detection_rules.inject_detection_events`:
  `result = drain + logs; result = existing_realtime_logic(...)`.

**Lifecycle**

- `delete_scenario(sid)` removes both backlog files (`.jsonl` + `.idx.json`).
- Restart-safe: backlog state lives entirely on disk; the in-memory
  ring buffer reseeds itself the next time the scenario is opened.
- `_load_scenarios()` migrates older entries by defaulting
  `mode="realtime"`, `visibility="public"` (back-compat for any
  scenarios that pre-date this change).

### C.2 — Setup notes (auto-generated, visible in scenario detail)

Every time a scenario is created or its phases are edited, we
auto-generate a `setup_notes` block describing exactly which sources
the operator must configure for the scenario to play out fully — what
kind they are (pull / push / bus), the recommended endpoint or push
target, the required auth, and any notable options.

**Data model** — `setup_notes` on the scenario JSON, regenerated by
`_generate_setup_notes(phases)`:

```json
{
  "generated_at": "2026-06-12T08:55:00+00:00",
  "summary": "This scenario emits events across 3 sources over 4 hours.",
  "sources": [
    {
      "source": "okta",
      "kind": "pull",
      "endpoint": "GET /api/v1/logs",
      "auth": "SSWS <user API token>",
      "options": ["since=<ISO8601>", "limit=100"],
      "notes": "Configure an Okta Event Hook or a polling collector pointed at this endpoint with your per-user API token. Events for this scenario will be served at their backdated timestamps on the first poll."
    },
    {
      "source": "azure_platform",
      "kind": "push",
      "endpoint": "Kafka topic 'azure-platform-events'",
      "auth": "SASL_SSL with the credentials shown on the Push Profiles tab",
      "options": ["batch_size=100", "compression=lz4"],
      "notes": "Start a push profile for azure_platform from the Push Profiles tab — the scenario will inject backdated events into the outbound stream."
    }
  ]
}
```

The static per-source hint table lives in `attack_scenarios.py`
(`_SOURCE_SETUP_HINTS`). Adding a new simulated source means adding
one row — trivial and reviewable.

**UI** — collapsed-by-default **"Setup notes"** card at the top of the
scenario detail page (and on the launch confirmation modal). Expands
to show the table; per-row source name links to the existing
admin-side documentation page for that source when present.

### Test plan (TDD before any code)

- `tests/test_scenarios_historical.py`:
  - `pre_stage_historical_events` writes the expected total event
    count for a 3-phase scenario.
  - Every staged event's `_ts` falls inside its phase's window.
  - Backlog file lines are valid JSON and source-tagged correctly.
  - `drain_historical_backlog("okta")` only returns Okta events and
    advances only the Okta sidecar offset.
  - Drain honors the visibility predicate: a `private` scenario owned
    by Alice yields zero events to Bob's caller context, all events
    to Alice's, and all events to admin.
  - `delete_scenario` removes both backlog files.
  - **Regression**: realtime-mode launch creates no backlog files;
    `inject_detection_events` behaves bit-for-bit as before.
- `tests/test_scenarios_setup_notes.py`:
  - `create_scenario` with 3 phases on 3 different sources produces a
    `setup_notes` block whose `sources` list contains 3 entries,
    de-duplicated and sorted.
  - Editing phases (`update_scenario`) regenerates the block.
  - Unknown source ids (a typo) surface as a `notes` row explaining
    that no hint is registered, rather than silently dropping the
    phase from the table.
  - The block is present in the JSON returned by `get_scenario` and
    in the export (`export_scenario`) — but not in the on-disk
    `attack_scenarios.json` round-trip from a re-import (it's
    regenerated, not authoritative).

### Open decisions

- **Source `generate_one(ctx)` helper**: most source modules expose
  `_generate_*` privately. C.2 needs a stable public seam to mint one
  event for backlog pre-staging. We can either (a) lift each
  `_generate_*` to a public `generate_one`, or (b) keep using the
  existing `inject_detection_events` machinery and feed a synthetic
  one-element batch through it. Leaning toward (b) for v5.1 (smaller
  blast radius); revisit if any source can't produce a single event
  on demand.
- **Backlog drain cap**: `max_items=None` (all-at-once per
  source/caller) is the default. If a customer reports "the first
  pull is too big for our collector", we add a per-source
  `APIGENIE_SCN_DRAIN_MAX_<SOURCE>` env var; not in v5.1.

## v5.2 — Multi-factor authentication (TOTP)

Single feature, shipping next. Builds on v5.1's Fernet key chain — no
new key-management surface.

Target branch: `feat/v5.2-mfa` off `main`.

### Goals

- TOTP (RFC 6238 / Google Authenticator-compatible), SHA-1, 6 digits,
  30 s step, ±1 step window for clock skew.
- 10 single-use backup codes per user, generated at enrollment,
  surfaced once as a downloadable `.txt`, argon2id-hashed at rest.
- Server-stored seed, encrypted with Fernet (same key as the admin S1
  token from v5.1 Phase B).
- Step-up at **login** (not per-request): a session is either
  `mfa_verified=True` or middleware refuses everything outside the
  MFA whitelist.
- **Mandatory** for built-in admin and every `is_admin=1` DB user
  from day one. Three-state `APIGENIE_MFA_USER_POLICY` env var
  (`off | optional | mandatory`) for the user rollout; default
  `optional` in v5.2.0, flipped to `mandatory` in v5.3.0.

### Non-goals (deferred — see FUTURE)

- WebAuthn / passkeys.
- SMS or email OTP (would force SMTP back into the stack — explicitly
  decided against in v5.0).
- Per-action step-up (re-MFA before destructive ops).
- Trusted-device cookies / "remember this browser".

### Threat model

- A leaked login password (DB dump, phishing, reuse) must not yield
  account takeover.
- A leaked `data/` directory (Google Drive sync, EBS snapshot, backup)
  must not yield usable MFA seeds — same Fernet key chain protects
  them as the admin S1 token.
- The built-in admin password file is treated as compromisable; MFA
  keeps that scenario recoverable.
- Brute-forcing the 6-digit code is mitigated by per-account lockout
  (5 failures / 15 min).

### Storage model

**`users` table — six new columns**

```sql
ALTER TABLE users ADD COLUMN mfa_enrolled         INTEGER NOT NULL DEFAULT 0;
ALTER TABLE users ADD COLUMN mfa_secret           TEXT;             -- Fernet ciphertext of the base32 seed
ALTER TABLE users ADD COLUMN mfa_enrolled_iso     TEXT;
ALTER TABLE users ADD COLUMN mfa_last_used_step   INTEGER;          -- replay guard, last accepted 30-s step counter
ALTER TABLE users ADD COLUMN mfa_failed_attempts  INTEGER NOT NULL DEFAULT 0;
ALTER TABLE users ADD COLUMN mfa_locked_until_iso TEXT;
```

**`mfa_backup_codes` table — new**

```sql
CREATE TABLE mfa_backup_codes (
    user_id   TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    code_hash TEXT NOT NULL,                  -- argon2id hash of a single backup code
    used_iso  TEXT,                           -- NULL == still usable
    PRIMARY KEY (user_id, code_hash)
);
```

Exactly 10 rows per user at enrollment; rows are stamped with
`used_iso` on use, never deleted, so audit + count remain trivial.

**Built-in admin** lives outside `users`, so the same six fields persist
in a single Fernet-encrypted JSON blob: `data/admin_mfa.json`.

**Sessions** (`_sessions` in `admin.py`): add `mfa_verified: bool` +
`mfa_verified_at: float`. A session with `mfa_verified=False` is
allowed to hit only:

- `/health`
- `/{admin,portal}/login` (GET + POST)
- `/{admin,portal}/logout`
- `/{admin,portal}/mfa/verify` (GET + POST)
- `/{admin,portal}/mfa/enroll` (GET + POST + `/confirm`)

Everything else returns `303 → /{admin,portal}/mfa/{verify|enroll}`.

### Algorithm choices

- **TOTP-SHA1 / 6 digits / 30 s step / ±1 step window**: maximises
  compatibility (Google Authenticator, Microsoft Authenticator,
  1Password, Bitwarden, Authy, FreeOTP).
- **Replay guard**: persist `mfa_last_used_step`. Reject codes whose
  step ≤ `last_used_step`. Closes the "same code submitted twice
  inside the 90 s window" hole.
- **Backup codes**: 10 codes, 8 chars each, alphabet
  `ABCDEFGHJKLMNPQRSTUVWXYZ23456789` (no ambiguous `0OIl1`), grouped as
  `XXXX-XXXX`. Hashed with **argon2id** (same primitive `accounts.py`
  already uses for passwords — no new dep).
- **Lockout**: 5 failures in 15 min → 15-min lockout. Per-account,
  not per-IP — IPs are cheap, accounts are expensive.

### Login flow (state machine)

```
                        ┌─────────────┐
                        │   /login    │
                        └──────┬──────┘
                               │ POST username + password
                               ▼
                  ┌─────────────────────────┐
                  │ verify_login() — OK?    │
                  └──┬───────────────────┬──┘
                     │ no                │ yes
                     ▼                   ▼
              ┌────────────┐    ┌────────────────────┐
              │ 401 + back │    │ mint pre-session   │
              │ to /login  │    │ mfa_verified=False │
              └────────────┘    └────────┬───────────┘
                                         │
                       ┌─────────────────┴────────────────┐
                       │ mfa_enrolled?                    │
                       └──┬─────────────────────────────┬─┘
                          │ no                          │ yes
                          ▼                             ▼
                ┌──────────────────────┐   ┌──────────────────────┐
                │ /mfa/enroll          │   │ /mfa/verify          │
                │ (if policy demands)  │   │                      │
                └─────────┬────────────┘   └─────────┬────────────┘
                          │ POST confirm code        │ POST code OR backup
                          ▼                          ▼
                ┌──────────────────────────────────────────────┐
                │ persist mfa_secret (if enroll)               │
                │ flip session.mfa_verified = True             │
                │ redirect to /{admin,portal}/                 │
                └──────────────────────────────────────────────┘
```

### Enrollment flow

1. `GET /{admin,portal}/mfa/enroll` — generate a fresh 20-byte
   (160-bit) base32 seed, cache it on the **pre-session**
   (`_sessions[token]['mfa_pending_seed']`) — never persisted until
   confirmed. Render an inline-SVG QR encoding
   `otpauth://totp/ApiGenie:<username>?secret=<seed>&issuer=ApiGenie&algorithm=SHA1&digits=6&period=30`
   plus the seed as text for manual entry.

2. `POST /{admin,portal}/mfa/enroll/confirm` (body: `{code}`) —
   validate against the pending seed (±1 step). On success:
   `accounts.enroll_mfa(uid, seed)` persists
   `mfa_secret = crypto.encrypt(seed)`, generates 10 backup codes,
   hashes + stores them, returns the 10 plaintext codes in the JSON
   response. UI renders a one-shot **"Save your recovery codes"** page
   with a `Download recovery_codes.txt` button and a confirmation
   checkbox.

3. **Reset / re-enroll** (lost device):
   - `POST /admin/api/rbac/users/{uid}/mfa/reset` (admin-only) clears
     `mfa_secret`, `mfa_enrolled`, all backup codes. User is forced
     through enroll on next login.
   - `POST /admin/api/me/mfa/regenerate-backup` (self-service) wipes
     the 10 existing backup codes and issues a fresh batch — requires
     confirming with current TOTP first.

### Policy & rollout

Single env var, three states:

| `APIGENIE_MFA_USER_POLICY` | User accounts                              | Admin accounts | Default in |
| --- | --- | --- | --- |
| `off`               | MFA disabled everywhere (break-glass / dev) | MFA disabled   | — (never default) |
| `optional`          | Users see a "Set up MFA" card; can skip     | **Mandatory**  | **v5.2.0** |
| `mandatory`         | Forced enrollment on next login             | **Mandatory**  | **v5.3.0** |

Built-in admin and `is_admin=1` users are always mandatory regardless
of the env-var value — `off` does **not** disable admin MFA, only user
MFA. Prevents "I'll just set it to off" from being a quiet admin-bypass.

**`app.py` — middleware extension**

1. Resolve the session.
2. If `mfa_verified=False`, block any path outside the MFA whitelist

3. If `mfa_enrolled=False` but policy demands enrollment, force the
   enroll path even when `mfa_verified=True` (covers mid-session
   admin reset).

### Operational impact (when v5.2 ships)

- **All admins must enroll** at their next login. No grace period.
  Plan a 5-minute window to walk admins through scan + verify +
  downloading their 10 backup codes.
- **Recovery codes** are shown **once**. If a user closes the page
  without downloading, they must hit
  `/admin/api/me/mfa/regenerate-backup` (which requires a fresh TOTP
  code) to get a new batch. There is no "show me the codes again"
  endpoint by design.
- **Lost device, no backup codes** → an existing admin uses the RBAC
  tab to **Reset MFA** on that user, who then re-enrolls on next
  login. **If the only admin loses their device and codes**, the
  bootstrap CLI ships a `python -m mfa reset-builtin-admin`
  break-glass that wipes `data/admin_mfa.json` (requires shell access
  to the host).
- **Fernet key rotation** (rotating `APIGENIE_SECRET_KEY`) invalidates
  every encrypted MFA seed. Users land on the enroll page and must
  re-scan. Backup codes survive because they're argon2-hashed, not
  encrypted — but they're per-seed, so re-enroll resets them too.
  Document loudly in `docs/SECURITY.md`.

### Open decisions (parked until kickoff)

- **QR rendering**: vendor a 50-line QR encoder vs. add `segno` to
  `requirements.txt`. Segno is mature, MIT, single file — leaning
  toward "pin it" for maintainability.
- **Backup-code count**: 10 (Google standard) vs. 8 (GitHub). Default
  to 10.
- **TOTP digits**: 6 vs. 8. Default to 6 for universal compatibility;
  revisit if a customer asks.

## FUTURE — un-scheduled

Small or speculative items kept on the radar. Nothing here has a
target release.

### Authentication / authorization

- **WebAuthn / passkeys** as an alternative second factor (or first).
  Significant work (challenge issuance, attestation, COSE keys);
  worthwhile once TOTP MFA is in the field and we have telemetry on
  the lost-device rate.
- **"Remember this browser for 30 days"** — signed cookie + a new
  `mfa_trusted_devices` table. Pleasant UX in the lab, security
  trade-off in production; deferred until requested.
- **Per-action step-up** — re-prompt MFA before destructive operations
  (delete user, rotate Fernet key, export DB). Easy once v5.2 lands.

### Event mix (Phase 5 — shipped) stretch goals

- **"Preview distribution" button** — generate 1k events with the
  current mix, render a mini histogram in the admin pane. Useful for
  validating non-obvious weight changes before saving.
- **Detection-rule injection weight knob** —
  `detection_rules.inject_detection_events` is currently force-injected;
  give it the same on/off + weight UI as the other event types.

### Operational

- **CLI break-glass for built-in admin MFA** — already in the v5.2
  plan as `python -m mfa reset-builtin-admin`. Generalise to a
  proper `apigenie-admin` CLI covering password reset, MFA reset,
  key inspection (does it match `secret.key`?), and a dry-run schema
  migration check.
- **Audit log** for security-sensitive events: MFA enroll/reset/fail,
  S1 console override changes from the browser, admin "act-as" entries.
  Append-only JSONL under `data/audit/`; surface in a new admin tab.

### Documentation

- **`docs/SECURITY.md`** — does not yet exist. Should consolidate the
  threat model that's currently scattered across docstrings in
  `crypto.py`, `accounts.py`, and `s1_detection_library.py`. Owner of
  this doc becomes the de-facto threat-model maintainer.
