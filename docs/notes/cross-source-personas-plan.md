# Cross-source entity correlation (Scenario Personas)

User feedback (2026-06-25 00:24 IT): "When I run an attack scenario,
logs arrive from every source and detections fire, but the entities
across sources are not the same. Different users, different IPs,
different hostnames per source. What kind of attack is that?"

He's right. As of commit `c3c5ee2` on branch
`fix/s1-site-scoped-tokens`, scenarios produce **synchronised log
noise**, not a correlated attack story. An analyst opening five
alerts tagged with the same `attack.id` sees five different victims,
five different attacker IPs, five different hostnames. The whole
"this is one attacker working through MITRE phases" illusion
collapses on first inspection.

## Proposed design (3 layers, TDD)

### Layer 1 — Scenario Persona Bundle

When a scenario is created (or first started), auto-generate a
`personas` dict and persist it on the scenario record so every
phase that fires reads the same values.

```python
scenario["personas"] = {
    "victim_user": {
        "name":      "John Doe",
        "username":  "jdoe",
        "email":     "john.doe@acme-corp.com",
        "upn":       "john.doe@acme-corp.com",
        "object_id": "<stable uuid v4>",      # Entra ID / AD objectGUID shape
    },
    "victim_host": {
        "hostname":   "JDOE-LAPTOP-7",
        "ip":         "10.42.13.7",            # RFC 1918 internal
        "os":         "Windows 11",
        "agent_uuid": "<stable hex>",          # OCSF device.uid shape
    },
    "attacker": {
        "ip":      "185.220.101.42",           # well-known TOR-exit-ish
        "country": "RU",
        "email":   "billing@invoice-corp.bad",
        "domain":  "evilcorp.bad",
        "asn":     "AS9009",
    },
    "malicious": {
        "file_name": "Invoice_Q4.docm",
        "sha256":    "<stable>",
        "md5":       "<stable>",
        "process":   "powershell.exe",
        "cmd_line":  "-enc <b64-of-decoy-payload>",
    },
}
```

Editing UI optional in Phase D — auto-generated values are demo-
quality on their own. Power users can override for a vertical-
specific demo ("victim is the CFO of acme-bank").

### Layer 3 — Hook in `_create_temp_rule`

When the scenario engine creates the temporary detection rule for a
phase, splice the per-source projection into the rule's
`field_overrides` BEFORE persisting. The injection loop in
`detection_rules.py` then stamps every field automatically — no
changes needed downstream.

```python
def _create_temp_rule(phase, attack_id, scenario_id, total_seconds, phase_start):
    overrides = dict(phase.get("field_overrides", {}))
    overrides["attack.id"] = attack_id
    overrides["phase.id"]  = phase.get("phase_id", "unknown")

    # NEW — splice persona projection for this source.
    scenario = get_scenario(scenario_id)
    personas = scenario.get("personas", {})
    source   = phase.get("source", "")
    import sources as _sources_pkg
    projection = _sources_pkg.get_persona_projection(source) or {}
    for field_path, persona_key in projection.items():
        # Resolve dotted path against the persona bundle.
        value = _resolve_persona_path(personas, persona_key)
        if value is not None:
            # Phase-level explicit overrides win — power users can
            # still pin a value for a specific phase.
            overrides.setdefault(field_path, value)

    rule_data = {
        "name": f"[SCENARIO] {phase.get('name', 'phase')}",
        "source": source,
        ...
        "field_overrides": overrides,
        "_scenario_id": scenario_id,
        "_attack_id": attack_id,
    }
    ...
```

Helper `_resolve_persona_path(personas, "victim_user.email")` is a
trivial nested-dict walker. Returning `None` lets the loop fall
through to source defaults — the existing random data — so an
incomplete persona dict degrades gracefully instead of crashing.

### Layer 4 — UI surface (optional, can ship later)

* Scenario Detail page: surface the persona bundle as a read-only
  card so the operator can refer to it ("this attack targets John
  Doe — search S1 for jdoe and jdoe-laptop-7").
* Inline edit modal: free-form text fields per persona slot, plus a
  "regenerate" button that re-rolls every slot at once.
* On the Scenario Activity feed, prefix each injected event with the
  resolved persona summary ("victim=jdoe@..., attacker=185.220...").

## Tests to write (TDD red-first)

1. `tests/test_scenario_personas.py`
   * `create_scenario` auto-generates a `personas` dict
   * `personas` survive save/load round-trip
   * Manual override of a persona slot persists and isn't clobbered
     by a regenerate

2. `tests/test_persona_projection_sources.py`
   * Each source listed in `_WIRED_SOURCES` has a `PERSONA_PROJECTION`
   * Every persona slot path referenced in projections exists in the
     canonical persona schema (no typos like `victim_user.emai`)
   * `sources.get_persona_projection(source)` honors alias resolution
     (entra_id → azure_ad, defender → microsoft_defender)

3. `tests/test_temp_rule_persona_splicing.py`
   * `_create_temp_rule` writes the projected fields into
     `field_overrides`
   * Existing phase `field_overrides` win over projected values
   * Missing persona slots fall through cleanly without crashing
   * Cross-source test: two phases on different sources within the
     same scenario get the SAME `victim_user.email`

## Open questions for later

- Persona slots are static — should we model phase-relative state
  changes? (e.g. attacker IP changes between Initial Access and
  Lateral Movement when pivoting through a compromised host.) Maybe
  Phase 2 of this feature.
- Should we randomize the victim slot across the small (~50) pool of
  "demo people" in `generators._ACTORS`? Probably yes for variety
  across scenarios, no within a single scenario.
- Should we link personas to the asset-binding XDR Asset ID pool
  (Phase 2 of asset-binding-plan.md)? Strong yes — if the user's
  S1 tenant has a real asset named `JDOE-LAPTOP-7`, the persona's
  `victim_host.hostname` should match.
