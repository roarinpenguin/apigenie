# RBAC & User Profiles — Design Document

**Status:** Design / In progress
**Date:** 2026-06-05
**Related:** `docs/MULTI_USER_LOG_PROFILING.md` (provides the CallerContext + conditional-binding matching engine this design reuses)

---

## Problem Statement

ApiGenie today has two coarse, password-based portals (`/admin`, `/portal`) and all
configuration (log profiles, detection rules, log push, listeners, source bindings)
is global. We need **real multi-tenancy**: individually-registered users, each with
their own isolated customizations, governed by granular **role-based access control**,
while admins retain full superuser reach.

The defining requirement: a user's **pull-source responses must match their own
environment** — custom fields, detection injects, and custom entities from *their*
log profiles — even though every user hits the *same* shared source endpoint.

---

## Core Model

### Permissions (5 levels, per Category)

| Level | Meaning |
|-------|---------|
| **View** | See the objects in the category |
| **Create** | Add new objects |
| **Modify** | Edit existing objects |
| **Delete** | Remove objects |
| **Manage** | Operate: execute / start / stop / enable / disable |

### Categories (user-assignable functionalities)

`log_profiles`, `detection_rules`, `log_push`, `listeners`, `source_bindings`

**Admin-only (never in entitlements):** Intrusions, Investigations & Bans,
Container Logs, Observability, System Settings, Entitlement & User management.

### Entitlement

A named bundle mapping `category → [permissions]`. Assigned to a user account.

```json
{
  "id": "ent-...",
  "name": "Telemetry Engineer",
  "description": "Full control of own profiles + read-only rules",
  "permissions": {
    "log_profiles":   ["view", "create", "modify", "delete", "manage"],
    "detection_rules":["view"],
    "log_push":       ["view", "create", "modify", "manage"],
    "listeners":      ["view", "create", "manage"],
    "source_bindings":["view", "modify"]
  }
}
```

### User Account

`username`, `email`, password (PBKDF2-HMAC-SHA256, same scheme as the admin hash),
optional `entitlement_id`, avatar (250×250 circular PNG), `confirmed` flag, and
per-user **settings** (see below). The built-in **admin** is a superuser: implicit
`*` permissions on every category, no entitlement required, can log into **both**
portals.

### Per-user Settings (editable in the User Portal)

- **SentinelOne Console URL + API Token** — for pulling/enabling Detection Rules.
  *Hint shown in UI: the token must have at least View + Manage rights for STAR Rules.*
- **Per-source identifiers** — see Identifier Matching below.
- Their **log profiles**, **source↔profile bindings** (mix % + volume/req),
  **detection rules**, **log push profiles**, **custom listeners**.

## Identifier Matching (the key mechanism)

Pull sources remain **one shared endpoint per source**. The response is personalized
by the **credential the collector presents**:

```
Collector → GET /api/v1/logs   Authorization: Bearer <user-A-token>
                                         │
                  auth.py extracts the presented credential value(s)
                                         │
        accounts.match_user_by_identifier(value)  →  user A
                                         │
   profiles.get_context(source, user=A) → A's profile / bindings / custom
                                         │   entities / detection injects
                              response shaped for user A
```

- Each user registers, in User Settings, one or more **identifiers per source**:
  `(source, id_type, id_value)` where `id_type ∈ {bearer_token, tenant_id,
  client_id, api_key, basic_user, ...}`.
- Matching is exact on `id_value`; the first/most-specific match wins (reusing the
  specificity logic from `MULTI_USER_LOG_PROFILING.md`).
- If no identifier matches → fall back to the default/global (admin "public")
  behavior — keeps existing single-tenant demos working.
- **Source Details must show placeholders** (`<YOUR_API_TOKEN>`, `<YOUR_TENANT_ID>`,
  …) instead of the legacy sample tokens, since the real value is the user's own
  registered identifier.

## Ownership & Visibility

Every customizable object (profile, rule, push profile, listener, binding) gains:

- `owner_id` — the user (or admin) who created it.
- `visibility` — `private` (default) or `public`.

Rules:
- A user sees their **own** objects + all **public** objects.
- Public objects are **view/use only** for users (not modifiable); only admins
  (or the owner) can modify.
- Users can **view events** generated by public listeners / push profiles.
- Different users never see each other's private objects.
- Admin sees everything and can act on anything (user-switcher, below).

---

## Admin Experience

- Logs into **both** portals with admin credentials, unrestricted.
- In the **User Portal**, a **user-switcher** ("Viewing as: ▾ <user>") lets the
  admin inspect and modify any user's customizations, change their password, etc.
- Manages **Entitlements** and **User Accounts** in **System Settings**.
- Can mark any profile/rule/push/listener **public**.

---

## Storage

**SQLite** (stdlib `sqlite3`, no new dependency) at
`/var/lib/apigenie/apigenie.db` — inside the existing `./data` volume, so it
survives container recreation and keeps the stack superportable.

Tables (Phase 1): `entitlements`, `users`, `user_identifiers`, `account_tokens`.
Phase 2 adds `owner_id`/`visibility` columns to the existing JSON-backed object
stores (migrated to SQLite or annotated in place — see Phase 2).

Sessions remain the in-memory map in `admin.py`, extended to carry
`{user_id, username, role}`.

---

## Removals (superseded by this design)

- **Investigation password gate** — removed. Investigations/Bans are plain
  admin-only features now.
- **User-portal shared password** — removed. Replaced by real per-user accounts.

---

## No-SMTP Account Lifecycle

No mail server. The admin creates a user, and ApiGenie mints a one-time
**confirmation/recovery token** rendered as a link the admin hands to the user
out-of-band. The user opens the link to set/reset their password. Tokens live in
`account_tokens` with a short TTL.

---

## Avatars

Upload ≤ 1 MB; **Pillow** resizes and crops to a **250 px circle** (transparent
PNG) stored under `/var/lib/apigenie/avatars/<user_id>.png`.

---

## Other

- **Log Push** gains the same detection-rule customization logic as Pull Sources
  (rules can inject/trigger events in a push profile's emitted stream).
- **Log Push password protection removed** — governed by entitlements.

---

## Implementation Phases

### Phase 1 — Identity & RBAC core
1. `accounts.py`: SQLite layer — users, entitlements, identifiers, tokens,
   permission model, login verification, `has_permission()`.
2. Wire accounts into `admin.py` sessions; both portals authenticate against
   accounts (admin remains env/file-based superuser).
3. Retire investigation + user-portal passwords.
4. Admin Settings UI: Entitlements + User Accounts CRUD.

### Phase 2 — Data ownership & per-user responses
1. Add `owner_id` + `visibility` to profiles/rules/push/listeners/bindings;
   enforce isolation + public publishing.
2. Per-source **identifier matching** in `auth.py` → `profiles.get_context(source, user)`.
3. **Placeholders** in Source Details.
4. Admin **user-switcher** in the User Portal.

### Phase 3 — Polish
1. Avatars (Pillow).
2. Admin-handoff confirmation / recovery links.
3. Detection-rule customization for Log Push.

---

## Backward Compatibility

- Empty accounts DB → behaves like today (admin-only), responses fall back to the
  global/public profile, existing sample-token demos still work until a user
  registers identifiers.
- Existing JSON object stores are treated as **admin-owned public** during the
  Phase 2 migration.
